WHAT/WHY: a SEPARATE batch feature (one-image-per-click stays). Mutate a base
caption into N variants → render all at LOW settings → user eyeballs gallery →
picks winners → promote a winner to FULL quality via existing Remix. Engine's
only job = base caption → N mutated caption STRINGS → feed EXISTING pipeline.

REUSE (no new plumbing): BatchCoordinator.RunBatchAsync(IReadOnlyList<string>)
is public + file-picker-independent (the batch seam). Remix metadata read =
promote + multi-generation. V4JsonPrompt model / V4JsonPromptSerializer (byte-
canonical, tested) / V4JsonPromptValidator already exist and are reused.

LOCKED DECISIONS:
  - Algorithm = Interactive Evolutionary Computation (human IS the fitness fn).
  - ONE change per variant (strict (1+λ)); base included as "variant 0" ref.
  - Axis pinning: LOOK (style+ornament) ⟂ SCENE (bg+elements+bbox); vary one.
  - bbox = Gaussian N(0,σ), strength slider subtle/moderate/bold = 2/5/10% of 1000.
  - palette ops in HSL; categorical genes = seeded library draw; BlendStyle = LOOK crossover.
  - Multi-generation: "Mutate from this" on a winner → its caption = new base.
  - Library = user-editable JSON folder (.../mutation-library), seeded from botanist fixtures.
  - HARD: NO Python ever. Schema-validity via existing C# serializer+validator.
  - Slot tags: [JsonIgnore] Element.SlotTag, never serialized (ElementJsonConverter
    writes fixed keys — verified leak-proof); keyword-inference fallback (SlotTagger).
  - Flexible: operators behind ICaptionOperator, behavior in MutationRunConfig,
    tunables as named constants — refine after first renders without rearchitecting.

    every key + InvariantCulture value round-trips identically; readers (Remix,
    CivitaiMetaBuilder) untouched. FALLBACK to the old decode+re-encode path for
