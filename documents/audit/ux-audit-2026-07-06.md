# Emberforge — Whole-UI UX Audit (2026-07-06)

Scope: every XAML page under `ImageGenerator.MAUI/Presentation/Views/`, the shared
styling system (`Resources/Styles/Colors.xaml`, `Styles.xaml`), and the shared
"pinned-header / scrolling-workspace / pinned-ActionBar" layout convention. This audit was
triggered together with two specific layout regressions (gallery detail, structure editor);
those two are fixed and recorded in the "Fixed this pass" section.

Legend: **[FIXED]** applied this pass · **[REC]** recommendation left for the user to decide.

---

## Overall health

The UI is in good shape and internally consistent:

- **One design language.** Every page composes the same primitives — `Card` (rounded
  surface container), `CardTitle` / `FieldLabel` / `HelperText` typography, and the
  `ActionBar` pinned strip. Colors are almost entirely `AppThemeBinding` (light/dark aware)
  drawn from the `Colors.xaml` palette.
- **Consistent button semantics.** `ActionPrimary` / `PrimaryButton` = the single filled
  brand-purple primary per ActionBar; `SecondaryButton` (outlined) = secondary actions;
  `DangerButton` (red) = Cancel / Delete / remove. Verified: exactly one primary button per
  ActionBar on every page.
- **Accessibility hooks are thorough.** Every interactive control declares an `AutomationId`,
  and buttons/inputs carry `SemanticProperties.Description` where the label alone is
  ambiguous. Disabled visual states are defined on all named button styles.
- **Status color palette is WCAG-AA** (light/dark pairs for Success/Error/Warning/Info/
  Canceled).

No Critical or High UX defects. Findings below are the two regressions plus low-risk
consistency polish.

## Fixed this pass

### 1. [FIXED] Gallery image-detail — preview overflowed, metadata below the fold
`Views/GalleryItemDetailPage.xaml`. Regression from `f55d117`: the preview `Image` was
changed to `MaximumHeightRequest="560"` and left inside the middle `ScrollView`. Inside a
ScrollView, MAUI measures children with unbounded height, so the max cap is ignored and the
~1024px shell preview rendered full-height, consuming the viewport and pushing the metadata
card off-screen.
**Fix:** the image now lives in its own bounded star row (`RowDefinitions="Auto,*,Auto,Auto"`)
outside any ScrollView, so `AspectFit` is measured against a real bounded height and fits
between the header and action bar. The metadata card sits in an `Auto` row with the editor
capped at `MaximumHeightRequest="220"` so long captions scroll inside the editor instead of
growing the row. Image and metadata title are now both visible without page scrolling.

### 2. [FIXED] Structure editor — fixed-width inspector + clipped element fields
`Views/IdeogramStructureEditorPage.xaml(.cs)`.
- **2a** The responsive handler pinned the left inspector to a fixed `360px` (wide) /
  `320px` (medium) column, so editing fields never widened with the window. Changed
  `ApplyWideWorkspace` / `ApplyMediumWorkspace` to **proportional star columns**
  (wide `1.3* / 1.6* / 1.3*`, medium `1* / 1.4*`) with `MinimumWidthRequest` on the
  inspector (320) and output (340) panes so they scale up but never collapse too narrow.
- **2b** The selected-element **Description** and **Text to render** fields were single-line
  `Entry` controls (no wrap → long text scrolled off horizontally). Converted both to
  multi-line `Editor` with `AutoSize="TextChanges"` + `MinimumHeightRequest="60"`, matching
  the scene-description editors. They now wrap and grow to show the full text.

### 3. [FIXED] Hard-coded hex → palette tokens
`Views/IdeaToPromptPage.xaml` line 109: the reference-image preview background used literal
`Light=#F3F4F6, Dark=#1F2937` instead of the palette. Replaced with
`SurfaceLight` / `SurfaceDark`. A full grep confirmed this was the **only** stray color
literal in the views (the `#RRGGBB` strings in the structure editor are placeholder example
text, not colors).

### 4. [FIXED] Duplicated `StatusKind` color triggers consolidated
The same 4–5-way `StatusKind → TextColor` `DataTrigger` block was copy-pasted into the
page-level status labels of MainPage, MutationEnginePage, IdeogramStructureEditorPage, and
IdeaToPromptPage. Moved the full Success/Error/Warning/Info/Canceled trigger set **once**
into the shared `StatusLabel` style in `Styles.xaml` (added the `vm` namespace there). All
four labels already used `Style="{StaticResource StatusLabel}"`, so the visual result is
unchanged — pages that previously handled fewer states now recolor correctly for all of
them. The per-job-card status label in MainPage's `GenerationJob` template (3 states, no
shared style) was intentionally left as-is.
**Resolved — DRY and compiled.** A first cut emitted 5 advisory `XamlC XC0022` warnings
because a `DataTrigger` binding inside a ResourceDictionary style has no `x:DataType` scope
and fell back to reflection. Fixed by introducing a one-member interface
`IStatusOwner { StatusKind StatusKind { get; } }` (`Presentation/ViewModels/IStatusOwner.cs`),
implemented (declaration-only — the `[ObservableProperty]` already generates the member) by
the four page ViewModels (`GeneratorViewModel`, `MutationEngineViewModel`,
`IdeogramStructureEditorViewModel`, `IdeaToPromptViewModel`), and tagging the style's trigger
bindings `x:DataType=vm:IStatusOwner`. Result: one shared trigger block, compiled bindings,
zero warnings, 1286 tests green.

### 5. [FIXED] Only style-less button given an explicit style
An app-wide sweep found exactly one button without an explicit `Style`: MainPage's per-input
"×" remove overlay (line 196), which silently inherited the implicit primary-purple default.
Gave it `DangerButton` so the remove affordance reads as destructive red, consistent with
Delete/Cancel elsewhere. (User: revert if you prefer the neutral look — purely cosmetic.)

## Follow-up (iteration 2, same day — from user feedback + a concept mockup)

After the first pass the user confirmed gallery is fixed but the structure-editor edit fields
still clipped content, and shared a concept mockup for the left pane. Also flagged the
"Describe an idea" page as partly unfinished (dropdown source selector, blank results pane).

- **[FIXED] Editor fields now show full content.** The element Description/Text and the
  scene High-level/Background fields were `Editor AutoSize="TextChanges"`, which is
  unreliable on Windows MAUI for programmatically-bound text (stays at min height → looked
  clipped). Switched to fixed generous heights with internal scroll (`AutoSize="Disabled"`,
  `HeightRequest` 90–160), matching the concept's tall scrollable textarea.
- **[FIXED] Structure editor left pane → accordion.** The Description / Style / Elements
  cards are now collapsible sections (tap the header; chevron follows via
  `BoolToChevronConverter`, no VM state). Style starts collapsed (it's optional). Cuts the
  long inspector scroll. Handler: `OnSectionHeaderTapped` toggles the section body's
  `IsVisible`. (The concept's "Manual / Generate" segmented control and drag-to-reorder
  element rows were deferred — reorder needs new VM commands; revisit if wanted.)
- **[FIXED] "Describe an idea" — segmented source toggle.** Replaced the 2-item
  `Text`/`Image` dropdown with a segmented **Text | Image** pill (active segment filled via
  `IsTextSource`/`IsImageSource` DataTriggers; code-behind sets `SourceMode`). No VM change.
- **[FIXED] "Describe an idea" — results empty-state.** Added a "Your prompt will appear
  here" placeholder card so the right pane isn't blank before the first Build.
- **[FIXED] "Describe an idea" — moved the "Also build Ideogram V4 JSON" checkbox** out of
  the source card and next to the prompt-writer, where output format belongs.

## Recommendations (report only — not applied)

These are lower-value or subjective; listed for the user to decide.

- **[REC] Duplicated converter declarations per page.** Each page re-declares the same
  converters (`NonEmptyStringToBool`, `InverseBool`, etc.) in its `ContentPage.Resources`.
  These could be promoted to app-level resources in `App.xaml` (two are already there) to
  cut boilerplate. Low risk but touches every page; deferred to avoid churn.
- **[REC] Extract a shared header control.** The `Grid ColumnDefinitions="Auto,*"` with a
  "← Back" `SecondaryButton` + `CardTitle` + `HelperText` subtitle is repeated verbatim on
  five pages. A small reusable `ContentView` (e.g. `PageHeader`) would DRY this up. Cosmetic/
  maintenance only.
- **[REC] GalleryPage and SettingsPage don't use the pinned-ActionBar pattern.** This is
  intentional (Settings auto-saves; Gallery's actions are per-tile), and both read fine — no
  change recommended, noted for completeness.
- **[REC] Element `Text to render` as multi-line.** Now an `Editor` (fixed this pass). If the
  literal render-text is expected to always be short, a single-line `Entry` with wider pane
  might feel tighter — revisit if the multi-line box looks oversized in practice.

All primaries are single-per-bar, all destructive actions use `DangerButton`, all controls
carry `AutomationId`s. No broken or unreachable actions found.
